# A 股投研助手产品定义

## 1. 项目目标

先做实用型工具，再逐步包装成简历项目。

项目核心不是做信息摘要，而是帮助用户从高噪声信息中形成可追踪、可解释的投资观点。

## 2. 产品定位

- 主入口是输入股票代码、股票名称或相关自然语言请求，直接做单票分析
- 辅入口是在用户没有明确目标股票时，推荐当下热门且值得进一步研究的候选股
- 保留统一检索能力，支持股票、主题、热门、普通问题的统一输入

## 3. 第一阶段交付形态

- 第一阶段先做 CLI
- 形态是交互式 Research Copilot，而不是单轮命令式工具
- 用户可以连续输入
- 系统自主判断当前背景是否充分
- 当背景不足、意图不清、候选过多或关键上下文缺失时，再触发结构化问询
- 后续第二阶段再把同一套后端能力包装成 Web

## 4. 总体架构方向

采用具有自主性的 Router-Orchestrator Multi-Agent 架构。

不是 plan-and-execute，而是更接近 ReAct / TAO 风格。

核心循环：

`Observe -> Think -> Act -> Update State -> Decide Next`

Orchestrator 需要根据当前状态自主决定下一步，包括：

- 是否继续检索
- 是否调用其他 agent
- 是否补证据
- 是否降级输出
- 是否向用户发起澄清问询

## 5. 系统角色划分

系统包含以下 Agent：

- Intent Router Agent
- Orchestrator Agent
- Hotlist Discovery Agent
- Stock Research Agent
- Search Agent
- Synthesis Agent
- Evaluator / Trace Agent

## 6. Clarification Gate 设计

Clarification Gate 是系统核心能力之一。

设计原则：

- 采用分层策略
- 普通问题尽量少打断，优先先做再说
- 单票分析、热门候选筛选、标的歧义、关键背景缺失、候选空间过大时，更愿意追问

触发机制：

- 标的歧义
- 任务目标不清
- 关键背景缺失
- 候选空间过大

交互风格：

- 一次只问一个问题
- 每次提供 2-3 个推荐选项
- 允许自由输入
- 表面上是在问用户，实际上是在帮助用户收敛问题和思路
- 系统给出的选项应体现推荐的分析路径，而不是机械索取参数

## 7. 记忆系统设计

采用双层记忆设计。

### 7.1 底层记忆

由系统自动维护，主要是工作记忆和过程记忆。

包括：

- 当前会话里的工作状态
- 当前分析对象、候选池、已确认约束
- 当前证据是否充分
- 已调用过哪些 agent
- 证据链、反证、推断与未解决问题

### 7.2 上层记忆

做成用户可见、可编辑的长期记忆文档，形式上类似 Claude 官网的个人身份记忆文档，但内容贴合投研场景。

建议包含：

- 投资偏好
- 风险画像
- 关注池
- 分析习惯
- 禁忌或约束
- 长期跟踪假设

记忆原则：

- 根据股票与金融行业的特性，只存储必要信息
- 不把整段历史对话机械塞入模型
- 通过结构化记忆提炼长期有效背景

## 8. 热门候选股定位

候选股发现偏“当下热门”，不是纯主题相关。

目标不是找最热，而是从最热里筛出最值得研究的。

采用混合热度驱动：

- 先看市场表现和交易热度
- 再结合新闻、公告、事件和叙事解释“为什么现在热”

发现链路分两阶段：

### 8.1 Heat Scan

用于识别市场正在交易什么。

参考信号：

- 价格与强度信号
- 资金与交易信号
- 事件与信息信号
- 叙事聚合信号

### 8.2 Research Filter

从高热度标的中筛出值得进一步研究的候选。

过滤维度：

- 是否有明确事件或叙事支撑
- 个股在主线中的地位
- 是龙头、补涨、跟风还是噪声
- 逻辑是否可持续
- 风险是否过于拥挤或过度情绪化

热门候选输出形态：

- 采用混合式展示
- 先给 3-5 个当前最值得看的候选
- 每只配一句“为什么现在值得看”
- 再补一层这些候选分别属于哪条市场主线

## 9. 单票分析定位

第一阶段 MVP 优先打磨单票分析闭环。

这是主入口，也是最核心的实用价值所在。

默认分析框架采用双视角：

- 交易视角：市场为什么此刻在交易它
- 投研视角：这个逻辑本身是否成立

整体上以投研视角为主，交易视角为辅。

重点服务的研究任务：

- 以中短线事件驱动为主
- 再兼容一部分热点博弈

## 10. 单票分析输出框架

输出风格采用结论优先。

同时采用双层卡设计：

- 默认展示极简卡
- 用户追问或系统判断有必要时，再展开完整卡

### 10.1 默认极简卡

- 观点倾向
- 一句话判断
- 当前价位
- 预期价位或区间
- 判断期限
- 置信度
- 市场主叙事
- 核心驱动
- 主要风险
- 观察点

### 10.2 完整观点卡

- 观点倾向
- 市场主叙事
- 核心驱动
- 主要风险
- 分歧点
- 观察点
- 证据链
- 信息变化
- 当前价位
- 预期价位或区间
- 判断期限
- 置信度
- 锚定依据

## 11. 观点倾向与价格框架

单票分析的核心判断采用：

- 看多
- 中性
- 看空

但判断不能脱离价格与时间。

必须建立在以下要素之上：

- 当前价位
- 预期价位或预期区间
- 明确判断期限

预期价位的生成采用混合锚定：

- 先用事件驱动逻辑判断方向和潜在空间
- 再用技术或交易锚点约束预期区间

参考锚定维度：

- 事件驱动重估
- 主叙事强化程度
- 预期差是否仍存在
- 前高、箱体、突破位
- 近期量价结构
- 相对板块强弱

时间维度：

- 由系统自主判断
- 但必须显式给出期限

置信度表达：

- 采用轻量显式方式
- 第一版建议使用低 / 中 / 高

## 12. Router 意图设计

Router 采用混合型意图划分。

先判断有没有明确标的，再判断用户要做什么。

第一版核心 intent：

- `single_stock_analysis`
- `stock_compare_or_followup`
- `hot_candidate_discovery`
- `theme_or_topic_exploration`
- `general_market_question`
- `clarification_required`

## 13. 状态展示与后台记录

前台采用轻量状态展示，后台记录完整过程。

### 13.1 前台状态展示

只展示简短状态提示，例如：

- 正在确认分析对象
- 正在收集近期事件与市场表现
- 正在补充公告/新闻证据
- 信息还不够，我想先确认一个问题
- 正在汇总观点与分歧

### 13.2 后台 Trace

必须记录完整过程，至少包括：

- 当前 intent
- intent 的判断原因
- clarification 是否触发及原因
- 每一步调用了哪个 agent
- 收集到了哪些证据
- 哪些证据支持主叙事
- 哪些证据削弱主叙事
- 哪些地方属于推断
- 为什么最终输出该观点倾向、价位区间、期限和置信度

## 14. 数据策略

第一版采用混合优先级数据策略。

- 热门发现链路优先依赖行情异动与热度信号
- 单票研究链路优先依赖公告、新闻、基础资料和市场表现

整体目标是先让“发现 -> 归因 -> 验证 -> 输出”两条链路都能跑通。

## 15. 对外叙事

项目对外表达采用“外显产品，内核 Agent”的策略。

对用户呈现：

- 一个会追问、会记忆、能输出观点卡的 A 股投研 Copilot

对架构和 README 呈现：

- 一个由 Router-Orchestrator、Clarification Gate、Memory、Trace 和多 Agent 协同驱动的自治研究系统

## 16. 当前已确认的 MVP 定义

一句话定义：

一个交互式 A 股投研 CLI Copilot，面向中短线事件驱动研究；用户输入股票代码、名称或自然语言请求后，系统通过 Router-Orchestrator 多 Agent 自主推进分析，在必要时用结构化问询帮助用户收敛问题，最终输出结论优先的双层观点卡，给出看多 / 中性 / 看空判断、当前价位与预期价位对比、明确期限、轻量置信度，以及可追踪的证据链和信息变化。

当前 MVP 的最小闭环：

- 聚焦单票分析闭环
- CLI 交互式会话
- 系统自主判断背景是否充分
- 必要时触发 Clarification Gate
- 多 Agent 补充证据
- 输出结构化观点卡
- 后台完整记录 trace
